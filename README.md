# promotion-figma-plugin
프로모션 기획안을 기반으로 와이어프레임을 생성하는 Figma 플러그인
# Promotion Concept Analyzer - Slide Link v3.4

Google Slides 링크 또는 PPTX 파일을 읽어 Figma 안에 프로모션 분석 결과와 기획안 기반 와이어프레임 목업을 생성하는 Figma 플러그인입니다.

이 프로젝트의 가장 중요한 목적은 기획안 슬라이드의 텍스트, 이미지, 도형, 버튼, 카드 영역을 분석해서 슬라이드별 구조와 위치 관계를 유지한 목업으로 가져오는 것입니다. 기획안 내용과 레이아웃은 계속 바뀔 수 있으므로, 특정 캡처 화면에 맞춘 고정 템플릿이나 예외 처리로 해결하지 않습니다.

## 핵심 원칙

- 기획안 슬라이드의 원본 좌표, 크기, 정렬, 그룹 구조를 우선 기준으로 사용합니다.
- 슬라이드마다 레이아웃이 다르면 결과물도 슬라이드별로 다르게 나와야 합니다.
- 상단바, 배지, 서브 타이틀, 타이틀, 본문, 카드, 버튼, 유의사항 텍스트는 원본 슬라이드의 위치와 위계 기준으로 분리합니다.
- 상단바에 있던 문구를 타이틀 영역으로 보내거나, 카드 안 문구를 본문처럼 재배치하지 않습니다.
- 텍스트만 추출해서 임의의 자동 레이아웃으로 다시 쌓는 방식은 최종 와이어프레임 결과로 사용하지 않습니다.
- 원본 좌표를 읽지 못하면 부정확한 목업을 만들지 말고, 사용자가 링크 권한이나 PPTX 업로드 상태를 확인할 수 있게 안내합니다.
- 디자인 코멘트, 수정 요청, 작업 지시 문구는 사용자 노출 목업에서 제외합니다.

## 포함 파일

- `manifest.json`: Figma 플러그인 이름, 실행 파일, 네트워크 접근 권한 설정
- `ui.html`: 링크 입력, PPTX 업로드, Pinterest 보드 입력, 실행 버튼, 상태 메시지 UI
- `code.js`: Figma 캔버스 생성, 분석 결과 렌더링, 와이어프레임 목업 렌더링
- `google-apps-script-proxy.gs`: Google Slides의 구조 데이터를 읽기 위한 Apps Script 프록시
- `slides-proxy-worker.js`: 공개 Slides/PPTX 요청을 보조하는 Cloudflare Worker 프록시
- `src/output/wireframe-output-policy.json`: 아웃풋 전용 설정 원본
- `scripts/sync-output-policy.js`: 아웃풋 설정을 `ui.html`, `code.js`에 동기화하는 스크립트
- `README.md`: 플러그인 목적, 실행 방법, 출력 기준, 유지보수 규칙

## 작업 영역 분리

Figma 플러그인은 최종 실행 파일로 `ui.html`과 `code.js`를 사용합니다. 하지만 작업자가 여러 명일 때 입력/추출/아웃풋 코드가 섞이면 회귀가 반복되므로, 아웃풋 수정은 아래 파일에서만 시작합니다.

아웃풋만 수정할 때:

1. `src/output/wireframe-output-policy.json`을 수정합니다.
2. `node scripts/sync-output-policy.js`를 실행합니다.
3. `node wireframe-regression-tests.js`를 실행합니다.
4. 문제가 없을 때만 `ui.html`, `code.js` 변경분을 함께 반영합니다.

아웃풋 작업자가 건드리지 않는 파일:

- `google-apps-script-proxy.gs`
- `slides-proxy-worker.js`
- Google Slides 링크 처리 함수
- PPTX XML 좌표 추출 함수

현재 `src/output/wireframe-output-policy.json`에서 분리한 항목:

- 와이어프레임 기준 폭
- 슬라이드 간격과 상하 여백
- 캔버스/placeholder 색상
- 이미지 목업 투명도
- 정확 모드/호환 모드 상태 문구
- 좌표 누락 안내 문구

## 실행 방법

1. Figma 데스크톱 앱을 엽니다.
2. `Plugins > Development > Import plugin from manifest...`를 선택합니다.
3. 이 폴더의 `manifest.json`을 선택합니다.
4. `Plugins > Development > Promotion Concept Analyzer - Slide Link v3.4`를 실행합니다.
5. Google Slides 링크를 입력하거나 PPTX 파일을 업로드합니다.
6. 필요하면 Pinterest 보드 링크를 함께 입력합니다.
7. `기획안 분석하기` 또는 `와이어프레임 만들기`를 실행합니다.

## 주요 기능

### 기획안 분석하기

Slides 또는 PPTX의 텍스트를 분석해 프로모션 정보와 디자인 컨셉 3안을 생성합니다.

생성 항목:

- 프로모션명
- 진행기간
- 목적
- 주요 스킴
- 명분
- 프로모션 인사이트
- 디자인 컨셉 3안
- 컨셉별 컬러 팔레트와 비율
- 메인 에셋, 텍스처, 구도, 배경 목업
- Pinterest 레퍼런스
- 타이포그래피 방향

Pinterest 보드 링크가 있으면 보드 이미지를 참고해 컨셉별 레퍼런스를 배치합니다. 외부 이미지가 Figma에서 막힐 수 있으므로, 가능한 경우 이미지 URL과 원본 링크를 함께 유지합니다.

### 와이어프레임 만들기

기획안 슬라이드의 실제 구조 데이터를 기반으로 Figma에 페이지 목업을 생성합니다. 이 기능은 `기획안 분석하기`와 별도로 실행되며, 분석 결과 생성 없이 와이어프레임만 만들 수 있어야 합니다.

와이어프레임 생성 기준:

- `메인페이지`라고 적힌 슬라이드가 있으면 그 다음 슬라이드부터 생성합니다.
- `메인페이지` 기준 슬라이드가 없으면 읽어온 슬라이드 전체를 대상으로 합니다.
- 슬라이드 전체 좌표계를 기준으로 1080px 폭에 맞춰 비율 스케일링합니다.
- 슬라이드의 일부 내용 영역만 잘라서 가운데 정렬하지 않습니다.
- 텍스트와 이미지 영역의 x/y 위치, w/h 크기, z 순서를 유지합니다.
- 이미지 데이터가 있으면 같은 위치와 크기에 원본 이미지를 저채도 목업 톤으로 배치합니다.
- 이미지 데이터가 없으면 같은 위치와 크기에 회색 목업 영역으로 대체합니다.
- 배경, 장식 도형, 라인, 원형 요소도 가능한 범위에서 원본 위치를 유지합니다.

## 와이어프레임 재현 우선순위

1. Google Slides pageElements 또는 PPTX XML에서 실제 요소 데이터를 읽습니다.
2. 각 요소의 좌표, 크기, 텍스트, 이미지, 도형, 선, 폰트 크기, 정렬, 순서를 정규화합니다.
3. Figma에서는 원본 슬라이드 전체 좌표계를 그대로 스케일링해 배치합니다.
4. 공개 Slides SVG처럼 전체 슬라이드의 시각 구조가 있는 데이터는 보조 자료로 사용할 수 있습니다.
5. 텍스트만 있는 데이터는 프로모션 분석에는 사용할 수 있지만, 정확 모드 와이어프레임의 레이아웃 근거로 사용하지 않습니다.
6. 구버전 Apps Script처럼 좌표를 반환하지 않는 기존 환경에서는 오류로 멈추지 않도록 `compat-text-fallback` 호환 목업을 생성할 수 있습니다.

좌표를 읽지 못한 상태에서 생성된 호환 목업은 정확 좌표 기반 결과가 아닙니다. 최신 Apps Script 프록시 또는 PPTX 업로드에서 좌표/SVG/PPTX 구조가 들어오면 항상 그 데이터를 우선 사용합니다.

## 텍스트 위계와 폰트 규칙

기본 폰트는 `Spoqa Han Sans Neo`를 사용합니다.

고정 규칙:

- 배지, 딱지: 16px
- 서브 타이틀: 18px
- 타이틀: 28px
- 텍스트끼리의 기본 간격: 14px
- 텍스트와 이미지 사이의 기본 간격: 20px

나머지 텍스트는 기획안의 실제 위계에 맞춰 조정합니다.

- 본문은 타이틀보다 작게 유지합니다.
- 버튼 문구는 버튼 영역 안에 유지합니다.
- 카드 제목, 카드 보조 문구, 가격, 할인율, 수량 표기는 같은 카드 그룹 안에서 판단합니다.
- 유의사항은 작은 안내 문구로 유지합니다.
- 상단 네비게이션 문구는 상단바 영역에 유지합니다.

이 규칙은 원본 슬라이드 구조를 무시하고 강제로 재배치하기 위한 규칙이 아닙니다. 원본 좌표와 위계를 유지한 상태에서 목업 톤을 정리하기 위한 규칙입니다.

## Google Slides 링크 사용 조건

가장 안정적인 설정:

- Google Slides 공유 권한: `링크가 있는 모든 사용자 보기`
- 조직 보안 정책에서 외부 export 접근 허용
- 공개 Slides 링크 또는 접근 가능한 일반 편집 링크 사용

지원 링크:

- `/presentation/d/.../edit`
- `/presentation/d/.../view`
- `/presentation/d/e/.../pub`
- `/presentation/d/.../export/pptx`

링크 접근이 막히면 Google Slides에서 PPTX로 다운로드한 뒤 직접 업로드하는 방식이 가장 안정적입니다.

## Google Apps Script 프록시

Google Slides의 구조 데이터를 안정적으로 읽기 위한 권장 방식입니다.

1. [script.google.com](https://script.google.com)에 접속합니다.
2. `새 프로젝트`를 만듭니다.
3. `google-apps-script-proxy.gs` 내용을 `Code.gs`에 붙여 넣습니다.
4. `배포 > 새 배포`를 선택합니다.
5. 유형은 `웹 앱`으로 선택합니다.
6. 실행 계정은 `나`, 액세스 권한은 `링크가 있는 모든 사용자`로 설정합니다.
7. 배포 후 생성된 `/exec` 웹앱 URL을 복사합니다.
8. Figma 플러그인의 `고급 설정 > Slides 프록시 URL`에 붙여 넣습니다.

Apps Script를 배포한 Google 계정이 해당 슬라이드를 열 수 있어야 합니다. 접근 권한이 없거나 조직 정책으로 막히면 PPTX 업로드를 사용합니다.

## Cloudflare Worker 프록시

Cloudflare Worker는 공개 Slides/PPTX 요청을 보조하는 선택 기능입니다.

1. Cloudflare Dashboard에서 `Workers & Pages > Create Worker`를 선택합니다.
2. `slides-proxy-worker.js` 내용을 붙여 넣고 배포합니다.
3. 생성된 `https://...workers.dev` URL을 복사합니다.
4. Figma 플러그인의 `고급 설정 > Slides 프록시 URL`에 붙여 넣습니다.

Worker는 공개 export 요청 보조에 적합합니다. Google Slides의 pageElements 구조를 안정적으로 가져오려면 Apps Script 프록시가 더 적합합니다.

## 디자인 컨셉 아웃풋 기준

- 진행기간은 `20NN.NN.NN (요일) 10:30 - 20NN.NN.NN (요일) 23:59 *총 N일` 형태를 기준으로 정리합니다.
- 정보 카테고리는 16px Bold, 정보 내용은 16px Medium 기준으로 표현합니다.
- 전체 텍스트를 과도하게 Bold 처리하지 않고 핵심 문장만 강조합니다.
- 주요 스킴의 불필요한 기호와 접두 문자는 제거하고 단어 중심으로 정리합니다.
- 메인 에셋 워딩은 기획안의 핵심 메인 비주얼 워딩을 기준으로 통일합니다.
- 메인 에셋, 텍스처, 구도, 배경은 이미지 생성이 아니라 HTML/Figma 도형 기반 목업으로 제작합니다.
- Pinterest 레퍼런스는 컨셉, 컬러, 무드가 맞는 이미지를 우선 배치하고 원본 링크를 유지합니다.

## 유지보수 규칙

- 특정 캡처 화면에만 맞는 하드코딩을 추가하지 않습니다.
- 새 기획안이 들어와도 같은 분석 흐름으로 처리되도록 좌표 기반 로직을 우선 수정합니다.
- 텍스트를 다시 줄 세우는 방식으로 문제를 덮지 않습니다.
- `text-only` 원본 응답을 그대로 최종 와이어프레임 입력으로 넘기지 않습니다.
- 구버전 프록시에서는 오류로 멈추지 않도록 별도 `compat-text-fallback` 목업으로만 처리합니다.
- 상단바, 타이틀, 카드, 버튼, 유의사항처럼 역할이 다른 텍스트를 한 영역으로 섞지 않습니다.
- 원본 슬라이드의 전체 좌표계를 유지하고, 내용 영역만 crop하거나 recenter하지 않습니다.
- 이미지와 도형은 원본 위치와 크기를 유지한 목업 요소로 변환합니다.
- 오류가 발생할 때는 부정확한 결과물을 만들지 말고 원인을 확인할 수 있는 메시지를 보여줍니다.

## 회귀 방지 테스트

수정 후에는 최소한 아래 검증을 실행합니다.

```bash
node wireframe-regression-tests.js
node --check code.js
node --check slides-proxy-worker.js
node --check wireframe-regression-tests.js
node --check --input-type=commonjs < google-apps-script-proxy.gs
```

현재 회귀 테스트의 핵심 기준:

- 좌표가 있는 slideElements는 정상 렌더링 입력으로 통과합니다.
- 공개 SVG 슬라이드는 전체 슬라이드 구조로 유지합니다.
- 텍스트-only 원본 데이터는 그대로 와이어프레임을 만들지 않습니다.
- 구버전 프록시 text-only 응답은 오류 대신 호환 목업으로 생성합니다.
- 버튼, 카드, 상단바 텍스트는 원본 좌표 그룹 안에 남아 있어야 합니다.
- 이미지 데이터가 없어도 같은 위치와 크기의 placeholder를 유지합니다.

## 제한 사항

- Google 또는 조직 보안 정책이 Slides export를 막으면 링크만으로 구조 데이터를 가져올 수 없습니다.
- 슬라이드가 통이미지 또는 스크린샷만으로 구성된 경우 텍스트와 요소 좌표를 정확히 분리하기 어렵습니다.
- 외부 이미지 URL은 Figma 환경에서 CORS 또는 원본 서버 정책 때문에 로드되지 않을 수 있습니다.
- 이미 생성된 Figma 캔버스 결과물은 자동 갱신되지 않으므로 수정 후 플러그인을 다시 실행해야 합니다.

## 문제 확인 체크리스트

텍스트 위치가 기획안과 다르게 흐트러질 때:

- Google Slides 링크 권한이 `링크가 있는 모든 사용자 보기`인지 확인합니다.
- Apps Script 프록시 계정이 해당 슬라이드를 열 수 있는지 확인합니다.
- 링크 접근이 막히면 PPTX 업로드로 다시 시도합니다.
- 좌표가 없는 텍스트 추출 결과만으로 와이어프레임이 만들어지고 있지 않은지 확인합니다.

이미지나 카드 구조가 다르게 나올 때:

- 원본 슬라이드가 통이미지인지, 개별 요소로 구성되어 있는지 확인합니다.
- PPTX로 다운로드했을 때 이미지와 도형이 분리되어 남아 있는지 확인합니다.
- 이미지 데이터가 막힌 경우에도 같은 위치와 크기의 목업 영역이 유지되어야 합니다.

## 담당 파일

- 분석 UI와 링크/PPTX 처리: `ui.html`
- Figma 프레임 생성과 목업 렌더링: `code.js`
- Google Slides 구조 데이터 프록시: `google-apps-script-proxy.gs`
- 공개 export 보조 프록시: `slides-proxy-worker.js`
- 플러그인 이름과 권한: `manifest.json`

버전명을 바꾸거나 배포 기준이 달라지면 `manifest.json`의 `name`과 이 README의 버전을 함께 업데이트합니다.
